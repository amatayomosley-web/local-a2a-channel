# Protocol spec

Wire format, validation order, ledger conventions, and security model for local-a2a-channel.

## Versioning

This document specifies protocol version **1.0**. Future protocol versions will increment the major or minor and document migration paths.

## Wire format

### Ping payload

A ping is a single JSON object written to `PING_<peer>.tmp` (UTF-8, no BOM, no trailing newline):

```json
{
  "seq": 42,
  "ts": "2026-05-23T16:00:00Z",
  "ledger_hash": "a3f5b2c14e8d9f0a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4",
  "mac": "7b3c8e1f5a9d2c4b6e8f0a2c4e6b8d0f2a4c6e8b0d2f4a6c8e0b2d4f6a8c0e2"
}
```

Fields:

| Field | Type | Description |
|---|---|---|
| `seq` | uint64 | Monotonically increasing sender-side counter. Increments by 1 per new turn. |
| `ts` | string | ISO-8601 UTC timestamp in `YYYY-MM-DDTHH:MM:SSZ` format. |
| `ledger_hash` | string (hex) | SHA-256 of the newest turn's body in `ledger.md`, computed by sender at ping time. |
| `mac` | string (hex) | HMAC-SHA256 over the canonical message string. Key is the shared secret. |

### Canonical message string for HMAC

The message signed by HMAC is the literal three-field string concatenation with `|` separators:

```
<seq>|<ts>|<ledger_hash>
```

Example: `42|2026-05-23T16:00:00Z|a3f5b2c14e8d9f0a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4`

Notes:
- No newline at end
- `seq` rendered as decimal integer with no leading zeros
- `ts` rendered exactly as the JSON `ts` field
- `ledger_hash` rendered as lowercase hex with no leading `0x`

### Ledger format

Markdown file at an agreed path (e.g., `ledger.md`). Append-only with newest-at-top convention.

File header (optional, written once at file creation):

```markdown
# <Channel name>

> Description of the channel and its purpose.

---
```

Each turn:

```markdown
## Turn <N> — <Sender> → <Recipient> (<YYYY-MM-DD>)
**From:** <Sender identity>
**To:** <Recipient identity>
**Date:** <YYYY-MM-DD>
**Re:** <short subject line>

<Turn body — markdown prose, with fenced JSON blocks for structured payloads>

— <Sender>

---
```

- Turn numbers are monotonically increasing global integers (both sides share the same counter; the next author increments)
- Newest turn appears immediately after the file header, above all older turns
- `---` lines separate turns
- Empty line between header block and body, between body and closing signature, and between signature and `---`

### Ledger hash computation

The `ledger_hash` field in the ping is computed as:

1. Read `ledger.md` as UTF-8
2. Normalize line endings (`\r\n` → `\n`)
3. Split on regex `(?m)^---\r?$`
4. Take `parts[1]` (the newest turn — index 0 is the file header)
5. Strip leading/trailing whitespace
6. Compute SHA-256 of the resulting UTF-8 bytes
7. Render as lowercase hex string

This binds the ping to the ledger state the sender saw at ping time. Receivers compute the same hash and reject the ping on mismatch (defends against stale or replayed pings against a moved ledger).

## Validation order (receiver)

When a watcher detects a new or modified ping file, it MUST validate in this order. Failure at any step drops the ping silently (log only).

1. **Parse JSON.** Reject if malformed.

2. **Time window.** Compute `abs(now - payload.ts)`. Reject if exceeds 30 seconds. (Defends against ping replays after long delays and against clock-skewed senders.)

3. **Sequence monotonicity.** Read last-seen sequence from per-receiver state file. Reject if `payload.seq <= last_seen_seq`. (Defends against ping replays within the time window.)

4. **HMAC.** Recompute `HMAC-SHA256(secret, "<seq>|<ts>|<ledger_hash>")` using the shared secret. Compare constant-time against `payload.mac`. Reject on mismatch. (Defends against unsigned and forged pings.)

5. **Ledger hash.** Recompute `ledger_hash` per the algorithm above. Reject if `payload.ledger_hash != recomputed`. (Defends against pings against a stale ledger state.)

Only after all five pass: persist `payload.seq` to the receiver's last-seen state file, then fire the wake handler.

## Shared secret

The HMAC key is a shared secret — 32 random bytes generated by a CSPRNG.

### Storage

Each side stores its copy of the secret in a per-user encrypted store. Recommended implementations:

- **Windows:** DPAPI (`ProtectedData.Protect` with `DataProtectionScope.CurrentUser`)
- **macOS:** Keychain via `security` CLI or Python `keyring` library
- **Linux:** gnome-keyring / libsecret via Python `keyring` library

The encrypted blob is written to a file at an agreed path (e.g., `%LOCALAPPDATA%\local-a2a-channel\ping-secret.key`). The file is readable only by the user (set ACL to owner-only on creation).

### Bootstrap (first run)

The first agent to start uses an atomic-create approach:

1. Attempt to atomically create the secret file (file mode `CreateNew` / `O_CREAT|O_EXCL`)
2. If CreateNew succeeds: this side is the originator. Generate 32 CSPRNG bytes, encrypt, write to file, log the SHA-256 of the secret to the audit log
3. If CreateNew fails (file exists): the peer beat us to it. Read the existing file, decrypt, use it

Race-free because filesystem CreateNew is atomic at the OS level.

### Rotation (optional, recommended every 90 days)

1. Initiator generates new 32-byte secret, encrypts, writes to `ping-secret.key.new`
2. Other side detects `.new` file, validates (32-byte decode), replaces in-memory secret
3. Initiator (or other side after swap) overwrites `ping-secret.key` with `.new` content
4. Both sides reset their sequence counters to 0
5. Initiator deletes `.new` file

## Security model

### What this defends against

| Threat | Defense |
|---|---|
| Unauthorized local process triggering wakes | HMAC fails without the shared secret |
| Replay of old pings within 30s window | Sequence monotonicity |
| Replay of old pings beyond 30s window | Time skew check |
| Spoofed pings against a stale ledger | Ledger-hash binding |
| Single-bit ping payload tamper | HMAC over (seq, ts, ledger_hash) |

### What this does NOT defend against

| Threat | Why not |
|---|---|
| Attacker with same user account | DPAPI / keyring is per-user; user-level access = full access to secret |
| Adversarial peer | Cooperation protocol assumes both parties friendly. Use a zero-trust protocol if not. |
| Ledger tampering at rest | The ledger is plaintext. Use git or filesystem integrity tools if needed. |
| Process-memory dump of decrypted secret | The secret is plaintext in process memory during validation; an attacker with debug privileges can dump it. Out of scope. |
| Network attacker | This protocol has no network surface. Not a defense, just a non-feature. |

### Constant-time HMAC compare

Implementations MUST use a constant-time byte comparison for the HMAC verification step. Use language-native constant-time primitives:

- **Python:** `hmac.compare_digest()`
- **C# / PowerShell:** byte-by-byte XOR-accumulate, never short-circuit on first mismatch
- **Go:** `subtle.ConstantTimeCompare()`
- **Rust:** `subtle::ConstantTimeEq`

Short-circuiting string compares (e.g., PowerShell `-eq` on strings) leak timing information about which byte of the MAC mismatched and MUST NOT be used.

## Wake handlers

The wake handler is the agent-UI-specific glue invoked by the watcher when a ping validates (or polling fallback triggers). It signals the agent process that there's new content to read.

The protocol does NOT specify how the wake handler works — it's deployment-specific. Reference implementations are provided for:

- **Claude Desktop on Windows** (clipboard-paste via UI Automation; see `reference/wake_desktop.ps1.example`)
- **Antigravity IDE on Windows** (agentapi `send-message` CLI; see `reference/wake_antigravity.py.example`)

Other agent UIs need their own wake handler. The constraint is:
- The wake handler MUST carry only a canonical "go read the ledger" signal
- Content MUST NOT be passed via the wake handler — it lives in the ledger only
- This prevents side-channel injection of unsigned content

## Polling fallback

Edge-triggered ping delivery can fail (clock skew, lost mtime event, focus deadlock on a UI wake handler). The polling fallback is a periodic safety net.

Each watcher additionally polls the ledger every ~30 seconds (configurable). On each poll:

1. Parse newest turn from `ledger.md`
2. Extract turn number from `## Turn N — ` header
3. Read `**To:**` field; check if addressed to this agent
4. Compare turn number against `lastseen_turn` state file
5. If newer turn addressed to this agent, fire wake handler and update `lastseen_turn`

The state file MUST be updated by both the edge-trigger ping success path AND the polling fallback path, so the two mechanisms don't double-fire on the same turn.

## State files

Each side maintains a small set of state files at a per-receiver path (e.g., `%LOCALAPPDATA%\local-a2a-channel\`):

| File | Format | Purpose |
|---|---|---|
| `seq-<self>-to-<peer>.txt` | ASCII uint64 | Sender's own outgoing counter |
| `lastseen_<peer>.txt` | ASCII uint64 | Receiver's last-accepted peer sequence (replay defense) |
| `lastseen_turn.txt` | ASCII int | Receiver's last-processed ledger turn number (polling state) |

State files are per-side, not shared. They survive process restarts but reset on rotation.

## Version compatibility

A receiver implementing protocol 1.0 MUST:
- Reject pings with unknown fields (forward-incompatible payloads)
- Accept the field set above exactly

Future protocol versions extending the payload SHOULD use a `"version"` field and ensure the canonical message string is computed differently from 1.0 so old receivers reject new pings safely.
